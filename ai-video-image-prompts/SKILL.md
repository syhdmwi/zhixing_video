# AI Video Image Prompts

## Overview

这个 skill 只负责静帧生图提示词，不负责图生视频，也不负责数字人视频轨。目标是把一段文案或镜头表，直接转换成一组可复制、可批量执行、可直接喂给生图工具的图片提示词。

默认不要一拿到文案就直接生成 30 条正式提示词。标准路径应该是：

1. 收集必要参数
2. 识别会反复出现的人和物
3. 先把重复主体清单展示给用户确认
4. 用户确认后，默认直接分别生成这些主体的独立三视图给用户看
5. 三视图展示后，先让用户决定是否修改主体；如果要改，直接说明“修改哪个主体 + 修改内容”，确认无误后，再进入正式提示词批次
6. 再生成整批正式提示词
7. 用户确认提示词后，用这些提示词进入生图阶段

如果文案里存在会反复出现的人、动物、吉祥物、产品、关键物件，那么预览图确认和三视图参考图应视为默认步骤，而不是可选步骤。
提示词一旦确认，默认交给 `ai-video-prompt-to-images` 继续处理，不在本 skill 内重复重写提示词。

## Inputs

- 文案或镜头表
- 图片数量
- 图片比例
- 输出语言
- 用户上传的人物参考图
- 风格要求
- 风格预设
- 画幅
- 禁用元素
- 目标提示词数量

## Core Rules

- 重复出现的人、动物、吉祥物、产品、关键物件，都应被视为“重复主体”
- 如果某个重复主体本身是一组分工不同、会同时出现的角色，例如一组 AI 虚拟小人、多个岗位机器人、同体系不同工种角色，不应简单视为“同一个主体”，而应视为“主体族群”
- 重复主体应以用户参考图、用户描述或首次锁定设定为第一锚点
- 先写全局 `Subject Consistency Block`
- 再写全局 `Style Block`
- 最后写镜头局部变化块
- 不编造模型私有语法
- 每条提示词都要对应文案里的一个具体信息点或情绪节点
- 如果用户明确要求 `30` 个提示词，优先服从数量要求
- 如果用户未指定数量，优先跟随分镜数；没有分镜数时再按时长估算
- 先确认必要参数，再直接生成整批提示词
- 默认至少三分之一提示词是纯场景画面
- 默认剩余提示词以人物画面或人物参与画面为主
- 生成的提示词必须能直接用于生图工具，不要停留在概括层
- 如果多个镜头里反复出现同一个人或同一个物，也必须保持主体一致性
- 对于会反复出现的主体，先展示主体清单并让用户确认；用户确认后，默认直接分别生成每个主体的独立三视图给用户直观看
- 如果识别到的是“主体族群”，应先锁定统一的世界观与基础造型语言，再拆分为多个角色变体
- 不是所有重复名词都必须进入预览阶段，只有需要稳定视觉身份且会多次入镜的主体才必须先锁定
- 正式提示词的写法顺序，优先贴近 Google 官方 Imagen 提示词方法：先写主体，再写背景与环境，再写风格与修饰
- 文案里明确出现的热门品牌、大众熟知的 logo、典型地点、典型物品，可以直接进入正式提示词，作为场景元素、界面元素或识别元素使用
- 如果某个品牌、项目、平台本身就有强识别的 mascot、IP 形象或固定视觉吉祥物，例如小龙虾 logo、卡通动物形象、固定机器人形象，则应优先直接把该形象当作可复用主体处理，而不是只写成抽象品牌元素
- 如果文案中的主要主体在现实世界或公开传播中本来就已经有稳定、明显、被大众普遍认知的视觉形象，应优先参考它现有的公认形象去制作，不要随意重新设计
- 对于大众熟知的品牌、平台、城市地标、景区地标、社会常见标识、服装品牌、汽车品牌、设备品牌，默认按“明确识别版”处理，不要只写成抽象氛围
- 当用户希望观众一眼认出该对象时，提示词里应直接写明：`官方logo清晰可见`、`品牌识别明确`、`工作区标识清晰可见`、`地标特征明确可识别`
- 如果对象较新、较冷门、较垂直、容易歧义，或你无法稳定判断其视觉识别特征，不要硬猜；应优先提示用户补充参考图、参考链接或更明确的描述
- 如果主讲人物没有提供参考图，默认先按动漫人物形象生成主体预览图
- 用户不满意三视图或主体展示结果时，应先修改该阶段结果，不要急着进入 30 张正式图
- 如果连续多次修改预览图仍不满意，应主动询问用户是否提供参考图
- 只要当前生图结果明显不合格，必须先明确提醒用户该结果不合格，并停在当前阶段
- 在用户明确同意之前，不允许把不合格的图继续当作锚点、参考图，或继续进入下一步流程
- 当结果不合格时，默认动作是：指出不合格点 -> 提醒用户需要重生成或修改要求 -> 等用户决定
- 用户满意主体清单后，先生成主体三视图作为参考图
- 正式 30 张图应尽量以已确认主体图和三视图作为参考锚点
- 所有主体预览图默认使用白底
- 所有三视图参考图默认使用白底
- 正式分镜图里，只要画面包含人物，默认使用半身照或半身构图，不要生成全身照，除非用户明确要求全身
- 当用户没有完整风格描述时，先用数字选择方式询问是否使用“风格模版”
- 风格预设是标准入口之一，但允许用户在预设基础上继续微调
- 正式开始整批生图提示词前，应再问一次是否进入正常生图提示词生成
- 三视图展示后，固定引导用户直接说明“修改哪个主体 + 修改内容”，不要只笼统问是否修改

## Quantity Rules

提示词数量优先级：

1. 用户明确指定数量，例如 `30` 个
2. 按文案时长自动估算结果
3. 已有镜头表的镜头数

不要一边说按时长算，一边又忽略用户给的明确数量。

## Workflow

### 1. Ask The Required Intake Questions First

拿到文案后，先按这个顺序继续：

- 先收文案全文
- 使用风格模版请输入 `1`，自主设定风格请输入 `2`
- 如果使用风格模版，再同时确认风格模版、图片比例和生图模型

如果用户输入 `1`，再从 [references/style-presets.md](./references/style-presets.md) 展示可选项。

风格模版、图片比例和生图模型应放在同一轮确认，不要拆成多轮。

生图模型默认使用数字选择方式确认：

- 使用 `GPT-Image-2` 请输入 `1`
- 使用 `nanobanana-2` 请输入 `2`

如果用户明确选择：

- `GPT-Image-2`

后续提示词优先按 [references/gpt-image2-prompt-template-01.md](./references/gpt-image2-prompt-template-01.md) 的写法生成。

如果用户明确选择：

- `nanobanana-2`

后续提示词优先按 [references/nanobanana-prompt-template-01.md](./references/nanobanana-prompt-template-01.md) 的写法生成。

输出语言默认按 `中文` 处理。

只有当用户主动输入：

- `语言为英文`

才切换到英文输出；不要默认再单独追问一轮语言。

如果用户输入 `2`，则直接收集用户自己的风格描述，不再展示内置风格模版。

当用户选择 `自主设定风格` 时，必须优先按用户明确给出的要求执行，不要擅自回退到内置风格模版的默认视觉语言。

如果用户给了“风格参考图”，必须先分析这张图的：

- 整体画风
- 角色比例
- 光影气质
- 色彩倾向
- 画面完成度
- 科技感、卡通感、海报感等核心视觉特征

然后严格按用户的说明去继承这些“风格特征”。

如果用户明确说：

- 只参考风格
- 不参考里面的人物

则只能沿用这张图的整体视觉语言，不能沿用图中人物的长相、服装、发型、身份或角色设定。

如果用户同时给了人物三视图参考图或其他明确人物参考图，则人物外观必须直接以用户给的人物参考为最高优先级，不允许自行生成一个新的主讲人物来替代用户的参考。

关于图片数量：

- 如果用户已经明确指定数量，优先使用用户给的数量
- 如果用户没有指定数量，不额外追问，默认按文案时长自动估算
- 默认估算规则为每分钟 `8～10张`
- 默认按每分钟 `9张` 估算
- 低密度时按每分钟 `8张`
- 普通密度时按每分钟 `9张`
- 高密度时按每分钟 `10张`
- 默认先按普通密度处理
- 只有在信息点连续切换、品牌/事件/数字明显密集、转折频繁、几乎一句一镜时，才升级为高密度
- 不能只因为短句多或品牌名稍多，就直接判成高密度

文案时长估算默认参考总控里的规则：

- 中文：`230` 字/分钟
- 英文：`150` 词/分钟

不要在用户没主动给数量时，先多问一轮“要生成多少张图片”。

在问完这些基础参数后，如果当前任务是“新的文案继续做同系列内容”，还应优先问：

- 这次是否要使用指定的“人设与风格档案”

如果用户要复用档案，优先先读取档案中的人物锚点、重复物锚点、风格摘要和默认参数，再继续后续流程。

### 2. Identify Recurring Subjects First

先从文案里提取会反复出现的主体，例如：

- 主讲人物
- 红色卡通龙虾
- 品牌吉祥物
- 产品外观
- 固定设备或关键物件

如果这些主体会在多个镜头里反复出现，就不能跳过主体确认。

但要先做一次筛选，不要把所有重复名词都拉进预览阶段。默认只有这类对象必须先锁定：

- 会在多个镜头里持续出现，且外观需要保持一致的人物
- 会在多个镜头里持续出现的动物、吉祥物、IP 形象
- 会在多个镜头里持续出现，且有明确造型要求的产品、道具、装置

以下内容默认不单独做主体预览，除非用户明确要求它们也要固定形象：

- 平台名、品牌名、厂商名
- 抽象概念
- 普通 UI 界面元素
- 一次性出现的工具、图标、背景信息点

但如果这些内容本身就是大众熟知的视觉识别元素，且文案里明确出现，应允许在正式提示词里直接写出。例如：

- 微信 / 企业微信 logo 或界面识别元素
- 华为云、腾讯云、Coze、Kimi 等品牌识别元素
- OpenClaw 这类自带固定 mascot 的项目形象，例如小龙虾 logo 或小龙虾虚拟角色
- 大众熟知的地标、门店、设备外观、典型物件

这类元素默认不单独进入主体锁定流程，但不能在正式提示词里被无故省略。

例外规则：

- 如果该品牌或项目本身最强的识别点不是文字 logo，而是固定 mascot / IP 形象
- 并且文案里会反复提到它、拿它和其他对象对比、或要求观众一眼认出它

则应优先把它作为“重复主体”或“对照主体”处理，而不是只当作普通 logo 或界面元素。

通用记忆规则：

- 后续遇到新的文案时，只要出现这种“社会上本来就有明显形象的主体”
- 无论是品牌 mascot、项目 IP、动物形象、固定吉祥物、公众熟知角色，还是具有强公认外观的主体
- 都应默认优先沿用它已有的公认形象来制作形象图
- 不要把它当成全新角色重新发明

如果这些元素属于大众熟知对象，且用户想让观众一眼看出来是谁，不要只写成：

- 品牌元素
- 平台氛围
- 城市场景

而应优先写成：

- 官方 logo 清晰可见
- 工作区标识清晰可见
- 品牌识别明确
- 地标特征明确可识别

如果对象较新、冷门、容易歧义，或不是大众熟知对象，则不要靠猜测补全视觉特征。优先让用户：

- 上传参考图
- 提供参考链接
- 或明确描述该对象最重要的识别特征

完成识别后，不要立刻进入图像生成。必须先把识别结果作为“重复主体清单”展示给用户。

展示完主体清单后，先问用户：

- 直接按这份主体清单进入三视图生成请输入 `1`
- 需要补充或修改主体请输入 `2`

只有用户确认主体清单后，才进入下一步三视图生成。

### 3. Generate Subject Three Views First

在正式 30 张图之前，默认先为重复主体生成三视图阶段材料。至少包括：

- 人物三视图
- 重复物件或吉祥物三视图

如果存在多个重复主体：

- 每个主体都必须分开生成
- 每个主体都应有自己独立的一张三视图
- 不要把多个主体合成在同一张三视图里
- 可以并行提交多个三视图生图任务
- 但每个任务内部只能对应一个主体
- 允许“同时生成多个任务”，不允许“一个任务里混多个主体”

如果存在“主体族群”，例如：

- 一组 AI 虚拟小人
- 一组不同岗位的数字员工
- 一组同体系的机器人角色

则默认拆成两层：

- 一层是统一基础设定：同一世界观、同一材质语言、同一体型体系、同一设计家族感
- 一层是角色差异化：通过颜色、配件、职责标识、工具、姿态倾向区分不同角色

不要把整组角色都生成成完全一样的复制体。

默认应至少区分：

- 主控或指挥型
- 开发或执行型
- 分析或支援型

这一步的目标是让用户在一轮里直接看清主体长什么样，而不是只看文字设定。

三视图默认要求：

- 白色纯背景
- 单主体清晰展示
- 不做剧情场景
- 不加复杂环境元素
- 不出现任何文字、字母、水印、logo、排版元素

人物三视图默认规则：

- 如果用户已提供人物参考图，人物参考图优先级最高
- 如果用户没有提供人物参考图，主讲人物默认先按动漫人物形象生成
- 只有当用户明确要求写实、纪实、真人感时，才主动切换出动漫人物默认值

主体族群默认规则：

- 同族角色保持统一设计语言
- 但每个角色要有清晰区分
- 区分方式优先用颜色、胸牌、头部标识、工具、岗位配件
- 不要仅靠站位不同来区分角色

如果环境支持生图 API，就直接生成三视图；如果当前环境不支持直接调生图 API，就至少输出：

- 可直接用于三视图生成的提示词
- 三视图用途说明
- 后续要锁定的主体清单

### 4. Let The User Approve Or Revise The Subject Three Views

拿到三视图后，让用户确认：

- 人物是否满意
- 重复物件或吉祥物是否满意
- 是否需要调整配色、形状、气质、材质

展示完三视图后，固定提问：

- 需要修改主体请输入 `1`
- 不需要修改，直接进入正式提示词生成请输入 `2`

如果用户需要修改，应允许两种方式：

- 直接说出修改要求
- 上传参考图

如果用户不满意，先修改三视图阶段，不进入正式 30 张图。

如果同一主体连续多次三视图仍不满意，不要继续盲目重试，应询问用户是否可以提供参考图或更明确的视觉参考。

如果结果客观上已经不合格，例如：

- 多主体串图
- 本应单主体却出现多个主体
- 出现文字、水印、logo
- 主体跑形
- 与已确认参考图明显不一致

则即使用户还没来得及指出，也应先主动提醒用户当前结果不合格，并建议重生成。

### 5. Finalize Three-View References After Approval

用户确认通过后的三视图，直接作为后续正式出图的参考锚点，不需要再重复生成一轮新的三视图。

三视图默认要求：

- 白色纯背景
- 单主体清晰展示
- 不加剧情环境
- 不出现任何文字、字母、水印、logo、排版元素

### 6. Generate The Final Prompt Batch

在开始整批正式提示词之前，必须先经过主体三视图确认阶段。

只有当用户明确表示“不需要修改主体”时，才直接进入整批正式提示词生成。

三视图展示后的固定收口话术应为：

- 需要修改主体请输入 `1`，并直接说明“修改哪个主体 + 修改内容”，也可以上传参考图
- 不需要修改请输入 `2`

主体确认完成后，再根据文案生成整批正式提示词。提示词必须：

- 内容与文案逐段对应
- 可以直接复制到生图工具使用
- 统一整体风格
- 保持重复主体一致性
- 默认至少三分之一是纯场景画面
- 不要所有画面都有人物
- 不要所有画面都只是场景
- 有人物的画面和纯场景画面应交替或合理穿插，利于后续剪辑
- 白天和夜晚场景应合理穿插，不要整批都停留在同一时间氛围
- 室内和室外环境应合理穿插，不要整批都停留在同一空间类型
- 即使围绕同一个世界观，也要主动安排不同环境层次，避免单一场景造成审美疲劳
- 如果文案里是一组角色协作，不同角色应在不同镜头里承担不同职责，不要所有小人都长得一样、做同一件事
- 如果正式分镜图里同时出现多个主体，不要把多个主体的三视图参考图同时作为强参考挂进去；默认只给主主体挂强参考图，次主体优先用文字约束、品牌识别元素、工作区标识或弱参考控制，避免模型误判成角色展示页、设定板或三视图页

如果用户选择了风格预设，正式提示词应继承对应预设里的：

- 风格方向
- 色彩倾向
- 光影倾向
- 材质倾向
- 画面气质

例如当用户选择 `赛博朋克风格` 时，不应只写一句“赛博朋克风格”，而应继承该预设里的完整风格块、色彩倾向、光影倾向、环境倾向和负面约束建议。

如果用户在预设基础上又补了自己的描述，例如“赛博朋克但更偏粉蓝霓虹”“素描但更偏建筑草图”，则用户补充要求优先。

如果当前提示词策略要尽量贴近 Imagen 官方方法，单条提示词优先按以下顺序组织：

1. 主体（subject）
2. 背景与环境（background / context）
3. 风格（style）
4. 镜头 / 构图
5. 光线 / 时间氛围
6. 材质 / 细节
7. 负面约束

不要把提示词写成只有风格词和抽象氛围词，而主体与环境不明确。

如果文案明确提到热门品牌或大众熟知的标志性元素，正式提示词应在合适镜头里直接把这些元素写出来，不要因为它们没进入主体预览阶段就整体忽略。例如：

- 提到企业微信时，可直接写企业微信界面、企微风格工作台、企微 logo 元素
- 提到微信时，可直接写微信风格消息界面或微信识别元素
- 提到华为、腾讯云、Coze 时，可直接写对应品牌标识、平台界面或品牌识别视觉

这里的“重复主体一致性”包括但不限于：

- 同一个女生角色
- 同一个红色卡通龙虾
- 同一个品牌 mascot
- 同一个产品或设备外观
- 同一个固定道具

正式分镜图的人物构图默认要求：

- 有人物的正式画面，优先半身照、半身特写或胸像构图
- 不要默认生成全身站立图
- 让人物脸部、上半身、手部核心动作更容易看清
- 只有用户明确要求全身构图、走路构图、站姿全景时，才放开全身照

如果用户没有特别强调构图比例，默认使用：

- 人物/人物参与画面：约三分之二
- 纯场景画面：约三分之一

### 7. Let The User Confirm The Prompt Batch

整批提示词输出后，让用户确认：

- 提示词是否贴合文案
- 画风是否正确
- 人物与场景比例是否合适
- 是否可以直接用于生图

如果用户确认提示词没问题，下一步就是交给 `ai-video-prompt-to-images` 用这些提示词去生图。

### 8. Read The Full Script First

先完整理解文案的叙事结构，不要看到一句就立刻写一句。至少先识别：

- 开头钩子
- 问题提出
- 核心卖点或核心观点
- 案例或论证
- 转折
- 收尾 CTA

### 9. Allocate Prompt Slots Across The Script

把提示词数量分配到文案结构里，而不是平均切字数。优先保证：

- 开头钩子有独立画面
- 重要卖点有独立画面
- 数字、对比、案例、动作类句子优先独立成画面
- 结尾 CTA 有独立画面

### 10. Build One Global Prompt Bible

先固定：

- `Subject Consistency Block`
- `Style Block`
- `Negative Constraints`

再为每个提示词叠加局部镜头变化。

### 11. Convert Each Beat Into An Image Prompt

每条提示词都应回答这 4 个问题：

- 这条画面对应文案哪一段？
- 画面主信息是什么？
- 主角在做什么？
- 画面如何体现当前句子的情绪或结论？

### 12. Keep The Batch Cohesive

同一批提示词中，默认不应无理由改变：

- 主要人物的身份、脸部特征、发型、服装
- 重复动物或吉祥物的颜色、造型、表情体系、材质感
- 重复产品或物件的外观、配色、结构、识别特征
- 视觉风格
- 时代和世界观

## Prompt Pattern

每条提示词推荐结构：

```text
[Character Block from user reference]
[Recurring subject consistency block]
[Style Block from user preference]
[Shot-specific action and environment]
[Composition, lens, framing, lighting]
[Negative constraints]
```

如果用户提供的是文案而不是镜头表，则在写提示词前先补一个内部字段：

```text
[Narration beat]
[Visual goal]
[Prompt body]
```

## Model Routing Guidance

- 如果用户要更高一致性，优先强调参考图锚定和低变量镜头变化
- 如果用户自己有成熟模板，直接沿用其模板结构
- 如果用户没有成熟模板，输出通用结构，不发明厂商专属参数
- 如果文案里存在重复出现的非人物主体，必须像人物一样单独锁定一致性

## LLM Meta-Prompt Mode

如果任务目标是“帮用户写一个给大模型用的母提示词”，不要直接产出 30 条图片提示词，而是优先输出：

- 一段可直接粘贴给大模型的主提示词
- 目标输出格式
- 必填参数列表
- 一份简短使用说明

模板见 [references/master-prompt-template.md](./references/master-prompt-template.md)。

如果任务目标是“帮用户跑完整工作流”，默认顺序必须是：

1. 询问图片数量、比例、语言、风格
2. 识别重复主体
3. 先生成重复主体预览图
4. 用户确认或修改预览图
5. 生成主体三视图参考图
6. 再生成整批正式提示词
7. 默认保持至少三分之一纯场景
8. 让用户确认提示词
9. 用户确认后进入生图

## Script-To-Prompt Mapping Rules

优先把文案内容映射成以下类型的画面：

- 事实信息
- 情绪冲突
- 人物动作
- 环境变化
- 抽象概念的具象化表达

## References

- 风格预设： [references/style-presets.md](./references/style-presets.md)

避免大量重复这几类低信息密度画面：

- 连续多张只是人物站着说话
- 连续多张只是电脑界面或手机截图
- 没有动作、没有情绪、没有场景变化的空镜
- 与文案句意无关的炫技画面

详细规则见 [references/script-to-image-playbook.md](./references/script-to-image-playbook.md)。

## Output

如果直接输出图片提示词批次，每个条目至少产出：

- `shot_id`
- `narration_excerpt`
- `visual_goal`
- `frame_type`
- `model`
- `image_prompt`
- `negative_prompt_or_constraints`
- `consistency_note`

如果输出的是“给大模型的母提示词”，至少产出：

- `system_or_master_prompt`
- `input_variables`
- `expected_output_schema`
- `usage_note`

如果当前阶段是角色确认阶段，至少产出：

- `character_summary`
- `style_summary`
- `approval_questions`
- `revision_path_if_rejected`

如果当前阶段是三视图阶段，至少产出：

- `front_view_prompt`
- `side_view_prompt`
- `back_view_prompt`
- `consistency_lock_note`
- `render_notes`

- 如果当前阶段是主体预览阶段，至少产出：

- `subject_preview_list`
- `preview_prompts_or_preview_images`
- `approval_questions`
- `revision_targets`
